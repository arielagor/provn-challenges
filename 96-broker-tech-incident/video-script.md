INC-4471. The on-call engineer scaled the consumer from three pods to twelve at two AM and the lag drained, which is the right immediate move. But three things did not get resolved overnight. Duplicate shipment records are sitting in twelve broker TMS systems. Root cause is unconfirmed. There are no idempotency controls on the webhook delivery path, which means this happens again the next time a broker has a slow afternoon. My diagnosis is that the volume surge was an aggravating factor, not the cause. The cause is in the existing webhook-delivery service. The retry path recurses into processShipmentEvent on any broker failure, which re-fetches the broker subscription list and re-broadcasts to every broker on the load, including the ones that already received it. Under load, that recursion is the load amplifier that produced the forty-two-thousand message backlog. The fix is two things. Stop the recursion. Add idempotency keyed on Kafka coordinates plus broker ID, persisted to Redis.

Code walkthrough. The revised webhook-delivery service has six fixes. I will walk through them in the order they matter.

First, idempotency. I derive a key from topic, partition, offset, and broker ID. Topic and partition and offset are Kafka coordinates that are guaranteed unique and stable across consumer restarts. The broker ID is what makes the key per-recipient rather than per-event, so multiple brokers subscribed to the same load each get their own dedup slot. The key gets persisted via Redis SET-NX with a one-week TTL. If acquire returns false, this delivery already happened (or is in flight from a parallel pod), and the function returns without sending. That single check is what would have prevented the eight hundred forty-seven duplicate deliveries in INC-4471.

Second, the retry pattern. The original recursed into the top-level function. The new one is a bounded loop inside deliverToBroker. Five attempts, exponential backoff with full jitter, capped at eight seconds per backoff. Per-broker isolation is explicit. One broker's failure does not block the others, and the loop never re-broadcasts to brokers that already succeeded.

Third, error classification. Five-hundred series and 408 and 429 are retryable. Other 4xx is permanent and goes straight to the dead letter queue with the HTTP status preserved. Network errors and timeouts are transient and retry. The reason this matters: under the original code, a broker returning 401 because their auth token expired would generate the same retry storm as a broker returning 503 because they were overloaded. Different problems, same wrong response. The classification means we now stop retrying when retrying cannot help.

Fourth, structured logging. Every log line has eventId, brokerId, attempt count, latency in milliseconds, and the relevant status or error. The 2am on-call engineer can grep on either eventId or brokerId and narrow scope to the failure in one query. The original logger was a single string with concatenation, which is impossible to query usefully.

Fifth, parameterized SQL. The original query used template-string interpolation of the payload's loadId directly into the SQL. That is a SQL injection vulnerability. The fix uses parameterized binding. Constraint says I should call out at least one issue beyond the primary incident. This is that issue, and it is more serious than the lag spike because it is exploitable.

Sixth, payload validation with zod. The original called JSON.parse and used the result as if the schema were guaranteed. A malformed event would have crashed the consumer, which Kafka then retries, which is its own infinite-loop scenario. The new code validates against a schema, parks malformed events to the DLQ, and moves on.

The tests cover the four behaviors that matter for this incident. First-and-second-call dedup. Per-broker key isolation. Retryable versus permanent classification. SQL parameterization with adversarial input. Not exhaustive. Focused on the failure modes that produced INC-4471.

Runbook walkthrough. Two AM, pager fires. The runbook in section B1 says open three Datadog panels first. Consumer lag, webhook error rate, DLQ insertion rate. Threshold values are explicit, not vague. Sustained lag above five thousand. Error rate above five percent. DLQ insertion above the historical baseline of five per hour. If all three are below threshold, this is not the consumer lag incident, look elsewhere.

Then mitigation in order. The first step is not to scale up. The first step is to query the DLQ for which brokers are failing. If one or two brokers dominate, scaling does not help. Disable those brokers' subscriptions temporarily and the consumer recovers. If failures are spread across many brokers, then scale to twelve replicas, which is the consumer group's partition count, no point going higher. Watch the lag drain. If it does not drain, the bottleneck is downstream, page the platform team.

Resolution is not the same as mitigation. The runbook makes that distinction. Resolution requires lag below one thousand sustained for fifteen minutes, error rate below one percent, DLQ insertion back to baseline, and no broker calls in the last thirty minutes. INC-4471 was officially closed when the lag drained, but duplicates had already shipped. We learned that the hard way.

The closing question on the runbook. Was the surge handleable with the current architecture, or did the architecture amplify it? If the architecture amplified it, the post-mortem action is an architecture ticket, not alert tuning. That distinction is what prevents the same incident from recurring.

The AI moment. The first cut Claude produced kept the original recursive retry pattern. Just added idempotency around it. I rejected this. Adding idempotency to a recursive load-amplifier is a contradiction. The recursion is what generates the duplicate deliveries that the idempotency check is supposed to suppress. You can't fix the symptom without removing the cause. I rewrote the retry as a bounded loop inside deliverToBroker, with explicit attempt counter, backoff, and max attempts. Per-broker isolation is now explicit too, so a slow broker doesn't block the others on the same load.

The pattern I am watching for. AI assistants tend to add features without removing the underlying problem they were called in to fix. They patch on top. In production code, the right move is often to delete the broken pattern entirely, not to wrap it in safeguards. The recursive retry was the broken pattern. The fix is structural, not additive.
