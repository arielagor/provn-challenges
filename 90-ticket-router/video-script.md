I built a small Python script that reads support tickets from a JSON file, tags each one with a priority, and drops it into the right queue. The whole thing is one file, about 150 lines of real logic, and it runs from the command line.

Let me walk through the shape of it. The script has four small pieces. One function reads the JSON. One assigns a priority. One maps priority to queue name. One flags same-customer tickets that landed close together in time. Everything else is glue.

Priority is three buckets: HIGH, MEDIUM, LOW. The spec gives you an exact table, and I wrote the code to look like the table. Premium tier customers go HIGH. Any subject or description that contains one of five keywords goes HIGH. The keywords are urgent, cannot, broken, down, and failed. Everything else with a real description is MEDIUM. Tickets with no description at all are LOW. The routing is a simple dict: HIGH goes to urgent-queue, MEDIUM to standard-queue, LOW to backlog-queue. Three priorities, three queues. I did not add anything the spec did not ask for.

The one design choice I want to call out is rule ordering. Premium tier and keyword hits are checked before the no-description check. That matters, because HIGH is defined as an OR of two things, and neither of them requires a description. A premium customer who opens a ticket with no description is still HIGH. I got that wrong in a first draft and caught it by running the sample tickets.

On the mandatory AI question. I used Claude to plan and draft this. The first version it produced had five queue names. It invented a security-queue and a billing-queue from subject keywords. I went back to the spec and saw three queues, not five. Urgent-queue, standard-queue, backlog-queue. I told Claude to delete the extras. The same thing happened with field names: its first pass used priority_level instead of priority and had no file I/O at all. I sent it the spec directly and asked for a rewrite that matched every field name. That second pass is what shipped.

One thing I would do differently with more time. I would swap substring matching for word-boundary regex. Right now, if a customer writes the word candown in a sentence, it fires as HIGH because "down" is a substring. A word-boundary regex would match only real words. I left substring in because the spec says "contains" and I wanted to stay close to the wording, but in a real system I would want the stricter behavior and a small test matrix to prove it.
